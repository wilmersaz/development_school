🔗 APIs REST

Representational State Transfer - La arquitectura que cambió la web

📚 ¿Qué es REST?

Definición

REST (Representational State Transfer) es un estilo arquitectónico para diseñar servicios web. Fue definido por Roy Fielding en su tesis doctoral en el año 2000.

💡 En términos simples:

REST es un conjunto de reglas y principios que definen cómo debe comportarse una API web para ser eficiente, escalable y fácil de usar.

🎯 Objetivo principal:

Crear APIs que sean simples, uniformes y que aprovechen al máximo las características del protocolo HTTP.

🏗️ ¿Qué significa REST?

R
Representational

Los recursos se representan en diferentes formatos (JSON, XML, HTML)

S
State

El estado de la aplicación se transfiere con cada petición

T
Transfer

La información se transfiere entre cliente y servidor

🎯 Los 6 Principios de REST

1. 🏛️ Arquitectura Cliente-Servidor

Separación clara entre cliente y servidor. Cada uno tiene responsabilidades específicas.

Cliente: Interfaz de usuario, experiencia del usuario
Servidor: Almacenamiento de datos, lógica de negocio

2. 🚫 Sin Estado (Stateless)

Cada petición debe contener toda la información necesaria. El servidor no guarda información de sesión.

Ventaja: Mayor escalabilidad y confiabilidad
Ejemplo: Cada petición incluye el token de autenticación

3. 💾 Cacheable

Las respuestas deben indicar si pueden ser cacheadas o no para mejorar el rendimiento.

Headers importantes: Cache-Control, ETag, Last-Modified
Beneficio: Reduce la carga del servidor y mejora la velocidad

4. 🔄 Interfaz Uniforme

Uso consistente de métodos HTTP, URLs y formatos de respuesta en toda la API.

Incluye: Recursos identificables, manipulación mediante representaciones
Ejemplo: GET /users/123, POST /users, DELETE /users/123

5. 🏗️ Sistema de Capas

La arquitectura puede tener múltiples capas (proxies, gateways, firewalls) sin que el cliente lo sepa.

Capas típicas: Load balancer, API Gateway, Microservicios
Beneficio: Escalabilidad y seguridad mejoradas

6. 📝 Código bajo demanda (Opcional)

El servidor puede enviar código ejecutable al cliente (JavaScript, applets).

Nota: Este principio es opcional y raramente se implementa
Ejemplo: APIs que devuelven JavaScript para widgets

🎯 Recursos y URLs en REST

¿Qué es un Recurso?

Un recurso es cualquier información que puede ser nombrada: un documento, una imagen, un servicio, una colección de otros recursos, etc.

📋 Características de recursos:

  • • Tienen una URL única
  • • Pueden tener múltiples representaciones
  • • Se manipulan mediante métodos HTTP
  • • Pueden contener links a otros recursos

🌐 Ejemplos de recursos:

  • • Un usuario específico
  • • Una lista de productos
  • • Un pedido de compra
  • • La información de una empresa

Diseño de URLs REST

# ✅ URLs bien diseñadas
GET    /users           # Lista de usuarios
GET    /users/123       # Usuario específico
POST   /users           # Crear usuario
PUT    /users/123       # Actualizar usuario
DELETE /users/123       # Eliminar usuario

GET    /users/123/posts # Posts del usuario 123
GET    /posts?author=123 # Alternativa con query
✅ Buenas prácticas:
  • • Usar sustantivos, no verbos
  • • Plural para colecciones
  • • URLs jerárquicas
  • • Consistencia en naming
❌ Evitar:
  • • /getUsers o /createUser
  • • URLs con acciones
  • • Inconsistencias de naming
  • • URLs muy largas o complejas

📊 Códigos de Estado HTTP más usados

✅ Códigos 2xx - Éxito

200 OK GET, PUT

Petición exitosa con datos

201 Created POST

Recurso creado exitosamente

204 No Content DELETE

Exitoso pero sin contenido

ℹ️ Códigos 3xx - Redirección

301 Moved Permanente

Recurso movido permanentemente

304 Not Modified Cache

Recurso no modificado (cache)

❌ Códigos 4xx - Error del Cliente

400 Bad Request Datos inválidos

Petición malformada

401 Unauthorized Auth

Falta autenticación

404 Not Found No existe

Recurso no encontrado

⚠️ Códigos 5xx - Error del Servidor

500 Internal Error Servidor

Error interno del servidor

503 Service Unavailable Mantenimiento

Servicio no disponible

💻 Ejemplo: API REST de una Tienda Online

🏗️ Estructura de recursos

# Usuarios
GET   /api/users            # Lista usuarios
GET   /api/users/123        # Usuario específico
POST  /api/users            # Crear usuario
PUT   /api/users/123        # Actualizar usuario
DELETE /api/users/123        # Eliminar usuario

# Productos
GET   /api/products         # Lista productos
GET   /api/products/456     # Producto específico
POST  /api/products         # Crear producto
PUT   /api/products/456     # Actualizar producto

# Pedidos
GET   /api/orders           # Lista pedidos
GET   /api/orders/789       # Pedido específico
POST  /api/orders           # Crear pedido
GET   /api/users/123/orders # Pedidos del usuario

🌟 Ventajas de las APIs REST

🚀

Simplicidad

Fácil de entender y usar. Aprovecha HTTP que todos conocen.

📈

Escalabilidad

Sin estado permite escalar horizontalmente fácilmente.

🔄

Flexibilidad

Múltiples formatos de datos y fácil de cachear.

🌐

Interoperabilidad

Cualquier lenguaje puede consumir una API REST.

⚡

Performance

Soporte nativo para caché y compresión HTTP.

🔧

Mantenibilidad

Estructura uniforme facilita el mantenimiento.
